# InvEntropy 后端 API 问题汇总（自动化测试暴露）

## 测试概况
- 总计 79 个用例，通过 61 个，失败 15 个，异常 3 个
- 所有失败均为后端 API 问题，非测试框架问题

---

## 问题一：后端缺少输入参数校验（8个用例失败）

以下接口在收到无效入参时仍返回 code=200 和 msg="success"，没有进行前端校验依赖的后端兜底验证：

1. POST 提交项目 - 缺少 projectType 字段 → 期望 code=400，实际 200
2. POST 提交项目 - applicant 为空字符串 → 期望 code=400，实际 200
3. POST 提交项目 - amount 为空 → 期望 code=400，实际 200
4. POST 提交项目 - amount 为负数 → 期望 code=400，实际 200
5. POST 提交项目 - projectName 为空 → 期望 code=400，实际 200
6. POST 提交项目 - projectType 为无效值 → 期望 code=400，实际 200
7. POST 提交项目 - deadline 为过去时间 → 期望 code=400，实际 200
8. GET 获取项目时间线 - 传入无效的项目 ID → 期望 code=400，实际 200

根因：后端 Controller/Service 层缺少对必填字段、字段格式、业务规则的后端校验，直接入库或返回成功。

---

## 问题二：部分正常请求被后端拒绝返回 400（4个用例失败）

以下接口使用正常/合理的分页参数请求，预期成功返回 200，但实际返回 code=400 + msg="出错啦~"：

1. GET 获取项目列表 - 自定义分页参数查询（size=20, page=2）→ 期望 200，实际 400
2. GET 获取项目列表 - 页码为负数 → 期望 200（正常处理或返回空），实际 400
3. GET 获取项目列表 - 页码超出范围 → 期望 200 + 返回空列表，实际 400
4. POST 添加账号 - 正常添加账号 → 期望 code=200 + msg="success"，实际 400

根因：分页参数 size=20 可能超出了后端限制；对负数/超范围页码直接抛异常而非兜底处理；添加账号逻辑可能存在隐藏 BUG。

---

## 问题三：错误提示信息过于通用（3个用例失败）

后端返回的错误信息统一为 "出错啦~"，无法区分具体错误类型：

1. 添加账号 - 缺少用户名 → 期望 msg="添加失败"，实际 "出错啦~"
2. 添加账号 - 用户名重复 → 期望 msg="该账号已存在"，实际 "出错啦~"
3. 获取项目列表（多个边界场景）→ 期望 msg="success"，实际 "出错啦~"

根因：后端异常处理统一捕获后返回泛化的错误信息，未按具体错误场景给出差异化提示，前端无法据此展示有意义的用户提示。

---

## 问题四：分页查询边界行为异常（2个用例失败）

1. GET 获取项目审批列表 - page=0（最小值）→ 期望返回 page=1 + 5条记录，实际返回 page=0 + 0条记录
2. GET 获取项目列表 - 页码超出范围 → 期望返回空列表（records=[]），实际返回 400 报错

根因：分页参数未做最小值保护（page=0 未自动修正为 page=1），超范围页码未做兜底处理。

---

## 问题五：上传文件测试缺少测试文件（1个用例异常）

- 文件上传测试引用 `docx/项目1.docx` 但该文件不存在于项目中，导致 FileNotFoundError。需补充测试用的 docx 文件。

---

## 问题六：响应数据结构异常导致空指针（2个用例异常）

- 部分接口返回的 data 字段为 null/缺失，导致断言时出现 `TypeError: argument of type 'NoneType' is not iterable`。需排查对应接口在特定参数下返回 null 而非空对象的情况，统一返回空对象 {} 或 []。

---

## 修复优先级建议

| 优先级 | 问题 | 影响 |
|--------|------|------|
| P0 | 输入参数校验缺失 | 脏数据入库，安全风险 |
| P0 | 添加账号接口返回 400 | 核心功能可能已不可用 |
| P1 | 分页参数异常返回 400 | 列表页功能受影响 |
| P1 | 错误信息过于通用 | 用户体验差，调试困难 |
| P2 | 分页边界值处理 | 少数边界场景 |
| P3 | 上传测试文件缺失 | 不影响线上功能 |
